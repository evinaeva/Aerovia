# PlaneFlow — план реализации MVP

> Рабочий план под жёсткий MVP по **10 принципам** (см. конец [`../backlog.md`](../backlog.md)).
> Это не ТЗ — порядок и объём решает продукт-овнер. Опирается на анализ актуального
> кода (`index.html`, v0.20) и текущего состояния игры.
>
> **Важно (вводная от ПО):** дизайн/визуал будет меняться. Поэтому план делает упор
> на **механику и системы**, а не на конкретную отрисовку. Текущий «ночной радар» и
> спрайты считаем заменяемым слоем — см. раздел «Запас под смену дизайна».

---

## 0. TL;DR

**Хорошая новость: ~70% MVP уже в коде.** Готовы fuel/repair/board, VIP (частный
джет), medical, событие с бобром (лес), а также переиспользуемые паттерны —
«помеха на полосе → тап → выезд бригады → работа → возврат» и «спец-борт через флаги
в `spawnPlane`». Недостаёт по списку MVP трёх вещей: **день/ночь**, **дождь/снег** и
**снегоуборщик/де-айсинг**. Плюс нужно **свести разрозненное в один связный аэропорт**
и **выстроить кривую постепенного открытия механик**.

Целевой MVP (принцип 10): `1 биом · day/night · rain/snow · fuel · repair · VIP ·
medical · snowplow · beaver`.

| Фича MVP | Состояние | Где в коде |
| --- | --- | --- |
| Routing flow (ядро) | ✅ готово | `update()` зоны air→runway→field→bay; `down/move/up` |
| fuel / repair / board | ✅ готово | `SVC`, `LV.sides`, `spawnPlane` requests |
| VIP (частный джет) | ✅ есть (×2 деньги, ½ терпения) | `spawnPlane`, флаг `vip` |
| medical | ✅ есть (приоритет, высадка пациента) | `spawnPlane`, флаг `medical` |
| beaver / лес | ✅ есть (дерево/олень/птицы + бригады) | `FOR`, `updateForest`, `spawnHazard` |
| 1 биом | 🟡 есть `forest`, но параллельно живёт «кампания» без биома | `BIOMES`, `LEVELS` |
| day/night | ❌ нет цикла (весь арт — статичная «ночь») | — |
| rain/snow | ❌ нет (есть только `fog` как событие) | `K.FOG_*` |
| snowplow / de-icing | ❌ нет (арктика `ready:false`) | `BIOMES[1]` |

**Лишнее для MVP, что уже в коде** (по принципу 10 — «не плодить»): `emergency`
(топливо-на-нуле), `rush`/`wind`/`fog` как отдельные события, 10 уровней кампании,
5 «незрелых» биомов, 46 достижений. Это не выбрасываем, но **выводим за рамки MVP**
(см. §3).

**Ключевые переиспользуемые паттерны (экономят почти всю работу по новому):**
- **Бригады** (`crews`, `dispatchCrew`, фазы `out→work→home`) — готовый шаблон для
  **снегоуборщика и анти-айс машины**. Снег/лёд = новый `kind` помехи.
- **Спец-борты через флаги** в `spawnPlane` — шаблон для будущих типов без новой логики.
- **Планировщик событий** (`nextRush/nextWind/nextFog`, сравнение с `gameTime`) —
  шаблон для **смены погоды и цикла день/ночь**.
- **Конфиг уровня `events:{}`** + валидатор `validateLevels()` — готовая система
  **постепенного открытия механик** (принцип 6).

## 2. Принципы → как держим их в реализации

| # | Принцип | Что делаем в MVP |
| --- | --- | --- |
| 1 | core не убить (80/20) | События **не блокируют routing**: помеха закрывает 1 полосу из ≥3, бригада параллельна. Бюджет: ≤1 активной «спайс»-помехи на полосу, всегда ≥1 полоса открыта (уже соблюдается в `spawnHazard`). |

| 7 | satisfying feedback | Плавное скольжение бортов (есть), де-айс «облако пара», плуг сгребает снег, ковёр разворачивается. Список — §8. |
| 10 | жёсткий MVP | Ровно 9 фич из §0. Всё прочее — `parked` (§3). |

## 3. Заморозка скоупа (что НЕ в MVP)

Чтобы не нарушить принцип 10, явно паркуем (оставляем в коде/доках, но **выключаем**
из дефолтного MVP-прохождения):

- `emergency` (топливо-на-нуле) — пересекается с medical по роли «срочный»; убрать из
  стартовой кривой, вернуть позже.
- `rush` / `wind` / `fog` как отдельные события — заменяются единой системой погоды.
- 5 биомов кроме одного (`arctic/tropical/desert/mountain/megacity`) — оставить
  `ready:false`, не трогать.
- Глубина достижений (46), Daily/Weekly, косметика «коты-пилоты», ливреи, ангар,
  спецспособности, облачный сейв — **после** MVP.

> **Решено (ПО):** **единый аэропорт с погодой** — одна локация (лесной биом), где
> есть и лес у полос (бобёр/дерево/олень/птицы), и зимняя погода (день/ночь, дождь,
> снег→плуг). Реализовано: лесному биому выставлен `weather:true`. ✅

---

## 4. Шаг 1 — Свести в один аэропорт + декапл логики от арта

Цель: убрать развилку «кампания vs биом-карта», сделать **одну точку входа** в MVP-сессию.

1. Ввести единый `MVP_AIRPORT` (конфиг как у `LEVELS`/`BIOMES.level`) — поле, боксы
   fuel/repair/board, 3 полосы, сервисное здание сверху (точка выезда бригад уже есть —
   `serviceHome()`).
2. `levelEvents()` и `reset()` уже умеют разные источники событий — добавить ветку для
   `MVP_AIRPORT` с **погодой и помехами по флагам**, как у `forest`.
3. **Слой темы (под смену дизайна):** вынести цвета/толщины/эмодзи в один `THEME`-объект
   (сейчас частично `COL`, `FCOL`, `FOR`). Вся отрисовка читает оттуда — тогда новый
   дизайн = новый `THEME`, без правок логики.

Критерий готовности: запуск MVP-сессии из меню в один тап, старое поведение кампании
не сломано (валидатор + e2e зелёные).

---

## 5. Шаг 2 — Цикл день/ночь

> **Статус: логика готова** (дизайн-независимый слой). `dayCycle(time)` →
> `{phase, night}`; `nightAmount` считается каждый кадр и отдаётся в тест-API. Влияния
> на сложность нет. Осталась **визуальная часть** (градиент неба, огни) — за слоем
> дизайна, рендер просто читает `nightAmount`.

Чисто визуальный + лёгкий геймплейный нюанс, фоном (принцип 2 — не таймер ожидания).

1. `dayPhase` (0..1) растёт от `gameTime`; период ~90–120 сек на полный круг.
2. Отрисовка: интерполяция фона/освещения между «день» и «ночь» (ночью — огни ВПП и
   боксов ярче; это уже частично есть в «ночном радаре»). Всё через `THEME`.
3. Геймплей: опционально ночью чуть короче «видимость» только визуально, **без**
   изменения хитбоксов/баланса (чтобы не задеть core).
4. Звук: эмбиент-слой меняет тон день/ночь (опц., низкая сложность).

Критерий: плавная смена без рывков; не влияет на сложность; выключается тумблером.

## 6. Шаг 3 — Погода: дождь и снег + снегоуборщик/де-айсинг

Единая система погоды заменяет разрозненные `fog/wind`.

> **Статус: движок погоды готов** (логика). `weather ∈ {clear, rain, snow}`, планировщик
> окон по таймеру (`K.WEATHER_*`), эффект на руление (`weatherTaxiMult`, снег хуже
> дождя), **opt-in** флагом `weather:true` у уровня/биома. Покрыто валидатором и
> тестами. Осталось: визуал (частицы), **де-айсинг** (нужна зона/бокс — это уже
> решение по раскладке/дизайну) и **снегоуборщик** (помеха+бригада, §6.2–6.3).

1. **Планировщик погоды:** `weather ∈ {clear, rain, snow}`, окна по таймеру (паттерн
   `nextFog`). Дождь — преимущественно косметика + лёгкое замедление руления (как
   `FOG_TAXI`). Снег — несёт геймплей. ✅ **сделано (логика)**
2. **Снег → де-айсинг (новый routing-шаг, не таймер):** во время снега у борта перед
   вылетом появляется задача `deice`. Игрок ведёт борт в **отдельный бокс де-айсинга**.
   Усиливает core flow, а не ломает (принцип 1). ✅ **сделано** — решение ПО: отдельный
   бокс (инфраструктура у правого края поля, всегда открыт, без апгрейда); шаг `deice`
   впрыскивается перед `depart` в снегопад для обычных бортов (медицинский — мимо).
   Визуал бокса — плейсхолдер (цвет `--ice`, иконка-снежинка), под арт.
3. **Снег → снегоуборщик (помеха на полосе):** новый `kind:'snow'` в `spawnHazard` —
   полоса заносится, тап → выезжает **плуг** (`crews`, `neededCrew → 'plow'`), сгребает,
   полоса открыта. Полностью на готовом паттерне бригад. ✅ **сделано (логика)** —
   занос появляется только в снегопад, сам не расходится (нужен плуг), ≥1 полоса
   всегда открыта. Визуал плуга/заноса — плейсхолдер-эмодзи, под арт.
4. **Лёд на стоящем борту:** если борт ждёт под снегом слишком долго — де-айс
   «сбрасывается», нужно повторить (мягко, опц.). Это «следующая игрушка» (принцип 6).

Критерий: снег открывает 2 новых действия (de-ice борта, плуг полосы), всегда ≥1
полоса проходима, новых passive-таймеров нет.

## 7. Шаг 4 — Tiny stories: VIP · medical · beaver (полировка)

Механика готова — добиваем **сценки и фидбек** (принципы 5, 7).

1. **VIP:** при заезде в board-бокс — короткая сценка: лимузин подъезжает, **ковёр
   разворачивается** (анимация), вспышки фото. Растущий «rage meter» вместо простого
   ½-терпения — опц., если не усложняет.
2. **Medical:** ambulance подъезжает к board-боксу, носилки, борт получает приоритетную
   полосу (уже приоритетен). Добавить визуальную сценку высадки пациента.
3. **Beaver:** усилить читаемость — бобёр грызёт, дерево кренится, полоска «до падения»;
   при срубе бригадой — щепки/звук. Логика уже в `updateForest`.

Критерий: каждое событие читается силуэтом + даёт «ой что это» момент; не мешает core.

## 8. Шаг 5 — Satisfying feedback (сводный чеклист, принцип 7)

- Борты мягко скользят по сплайну — ✅ есть (`followPath`, `steer`).
- Рисование маршрута — плавный пунктир со свечением — ✅ есть.
- Машинки/бригады мило подъезжают и уезжают — ✅ паттерн есть, добить easing.
- VIP — ковёр разворачивается (новое).
- De-ice — облако пара/капель (новое).
- Плуг — сгребает снег с полосы (новое).
- Near-miss «уфф» + лёгкое слоумо — ✅ есть.
- Все звуки — мягкая пентатоника на событие — ✅ есть (`SND`).

## 9. Шаг 6 — Постепенное открытие (кривая анлоков, принцип 6)

Лесенка «всегда есть следующая игрушка». Реализуется через `events:{}` в конфиге
уровней/сессии (механизм уже есть и валидируется):

```
Блок 1: fuel + repair + board (чистый routing)
Блок 2: + rain (косметика, знакомство с погодой)
Блок 3: + VIP            (первая «tiny story»)
Блок 4: + snow + snowplow (де-айс + плуг)
Блок 5: + medical         (приоритетный борт)
Блок 6: + beaver          (лесная помеха)
```

День/ночь — фоном со старта (не «механика для открытия»). На каждый новый элемент —
тихий микро-туториал в стиле текущего (`tut`, «покажи и дай сделать»).

---

## 10. Шаг 7 — Тесты и валидатор (страховка)

Репозиторий ценит «страховку под механики» (`validateGame()`, `logic.test.mjs`, e2e).
Под каждый новый кусок:

1. Расширить `EVENT_KEYS`/`SVC_TYPES` (`deice`, `snow`-hazard, `plow`-crew) и
   `validateLevels()` — чтобы кривая анлоков §9 проверялась автоматически.
2. Юнит-тесты: погодный планировщик, де-айс как шаг routing, плуг открывает полосу,
   «всегда ≥1 полоса проходима».
3. e2e-смоук: запуск MVP-сессии, посадка борта при снеге + де-айс.

## 11. Запас под смену дизайна

Чтобы будущий редизайн не переписывал логику:

- **Единый `THEME`** (цвета, размеры, эмодзи/спрайты, толщины) — вся отрисовка читает
  оттуда; геймплей от него не зависит.
- **Разделение update/draw** уже соблюдается — держим его строго (никаких правок
  состояния в `draw*`).
- **Ассеты — заменяемый слой:** сейчас эмодзи/примитивы; новый арт подключается через
  тот же `THEME`/`drawIcon`/`planeShape`, не трогая `update`.
- Числа баланса — только в `K`/`FOR`/конфиге, не в коде логики (уже так).
