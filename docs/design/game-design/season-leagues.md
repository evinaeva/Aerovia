## ⚠️ Это НОВОЕ продуктовое решение (отход от «без сезонов») — ПОДТВЕРЖДЕНО 2026-07-01

Дизайн-док рейтингов фиксирует прежний выбор овнера:
[«ранг-медали — пороговые **НАВСЕГДА**, без сезонов/перевыдачи»](leaderboards.md#ранг-медали--пороговые-навсегда).
**Сезонная лига этот выбор нарушает** — по определению это повторяющееся окно с перевыдачей.
Как и сами рейтинги когда-то были сознательным отходом от «лёгкой меты», лига — это
осознанный шаг, который овнер должен подтвердить **до** реализации (см. «Открытые вопросы»).

Чтобы конфликт был минимальным, план **разводит два класса наград** (см. ниже): постоянные
пороговые медали остаются как есть, а «сезонным» становится только косметика + дивизион-бейджи,
которые тоже пороговые-навсегда. То есть модель «навсегда» мы не ломаем — добавляем поверх неё
ротацию контента.

## Зачем (урок кейса)

Главный вывод кейса — **не награды, а цифры**: у Subway Surfers Leagues **+45 %
игрового времени** у записавшихся, **75 % записавшихся — уже вовлечённые игроки**,
**+3 % инсталлов** от самого ивента, и — ключевое для нас — успех достигнут
*«минимальными усилиями разработки, переиспользовав уже отслеживаемые метрики»*
(у них — монеты и время; у нас — **обслуженные борта / combo / streak / earned**).

Единственный настоящий рычаг ретеншена здесь — **повторяющийся повод вернуться**:
«до конца сезона 3 дня, ты 2-й в Серебряном дивизионе». Всё остальное — обвязка вокруг него.

## Дизайн

### 1. Сезон = новый бакет `periodBucket`

Сейчас `periodBucket` отдаёт `all` / `YYYY-MM` / `YYYY-Www`. Добавляем **`season`**:
детерминированное 2-недельное окно UTC от фиксированного якоря.

```
SEASON_EPOCH = <фикс. понедельник 00:00 UTC>   // константа в конфиге
SEASON_LEN   = 14 * 24 * 3600 * 1000
seasonIndex(ts) = Math.floor((ts - SEASON_EPOCH) / SEASON_LEN)   // 0,1,2,…
ключ бакета     = 'S' + seasonIndex             // напр. 'S37'
номер для показа = seasonIndex + 1              // «Сезон 38»
```

Свойства сохраняем те же, что у существующих периодов: чистая функция, UTC, одинаковая
на клиенте и будущем сервере, срез = фильтр журнала по текущему бакету (без разрушающего
сброса — старые счета живут в all-time). Длину/якорь — в конфиг, чтобы менять без правки логики.

✅ **Сделано как описано** — `SEASON_EPOCH = Date.UTC(2026,0,5)` (понедельник),
`SEASON_LEN = 14 дней`, `seasonKey/seasonNumber/seasonDaysLeft` в
[`07-audio-services.ts`](../../../src/game/07-audio-services.ts); `periodBucket()` отдаёт
дополнительное поле `season`. ⚠️ **Одно отличие от плана:** `season` НЕ добавлен в массив
`PERIODS` (тот питает `ranks`/`onRank` — топ-100/топ-10/№1 «навсегда»), чтобы место в крошечном
сезонном бракете не подделывало глобальные ранг-медали; UI-вкладки берут отдельный список
`LB_TABS = PERIODS.concat(['season'])` в [`11-menu-ui.ts`](../../../src/game/11-menu-ui.ts).

### 2. Дивизионы (Bronze → Diamond)

Вместо одного глобального топа (где ты 4000-й) — **бракеты по силе**: мотивирует быть 2-м из 30.

- 5 дивизионов: 🥉 Bronze · 🥈 Silver · 🥇 Gold · 💠 Platinum · 💎 Diamond.
- **Назначение:** первый сезон игрока — по перцентилю его all-time `bestScore`; далее —
  **promotion/relegation** по итогу прошлого сезона (топ ~20 % ↑, низ ~20 % ↓).
- ⚠️ **Настоящие кросс-игроковые дивизионы без сервера невозможны** (та же причина, что
  глобальный топ — см. [`leaderboards.md`](leaderboards.md#статус-это-каркас-не-боевой-рейтинг)).
  На mock-провайдере дивизион — это честная **демонстрация** против сид-ботов
  ([`07-audio-services.ts` → `seedBots`](../../../src/game/07-audio-services.ts)): бракет и
  ботов подбираем под уровень игрока. Боевые дивизионы приходят с бэкендом (Фаза 3).

✅ **Сделано:** `seasonDivisionIndex(rank, total)` — перцентиль в сезонном топе (квинтили
0.2/0.4/0.6/0.8) для *первого* сезона игрока; `Leaderboard.season.standing(mode)` отдаёт
`{rank, total, divisionIdx, division, list, promoted, relegated}`. **Promotion/relegation
сделаны (вторым проходом, 2026-07-01):** `pf_season_division_v1` хранит последний
назначенный дивизион с его `seasonKey`; `resolveSeasonDivision(rec, curSeasonIdx, rank, total)` —
чистая функция: рекорд ЭТОГО сезона → дивизион зафиксирован (не пересчитывается повторными
заходами); рекорд ПРОШЛОГО сезона → `seasonPromote()` поднимает/опускает на 1 по тем же
квинтилям 0.2/0.8; пропущенные сезоны или первый заход → свежее назначение по перцентилю.
UI показывает «▲ Повышение! / ▼ Понижение» на вкладке «Сезон». ⏳ **Осталось:** подбор ботов
под уровень игрока (сейчас фиксированный список `seedBots`, как и у обычного рейтинга) — не
критично для честности демо, но упомянуто в исходном плане; настоящие бракеты по дивизиону
(не общий список) — ждёт бэкенда (Фаза 3).

### 3. Награды — два класса (мирим с «навсегда»)

| Класс | Что | Модель | Легенда? |
| --- | --- | --- | --- |
| **Постоянные бейджи дивизиона** | «Достиг Gold», «Достиг Diamond» и т.п. | пороговые **навсегда** (как ранг-медали) — флаг `comp:true` | **нет** (мимо «Легенды», как `comp:true`) |
| **Сезонная косметика** | 1 акцентный цвет за сезон (или за место) | **ротация**, хранится в отдельном реестре наград, не в `unlocked`-Set | н/д |
| (опц.) Мягкая валюта | бонус `money` за место | разовое начисление | н/д (не сделано) |

Ключевой приём: **`ACH.unlocked`-Set не трогаем ротацией** — туда попадают только
пороговые-навсегда бейджи дивизиона. Ротирующиеся призы идут в **отдельный
стор наград** (`pf_season_rewards_v1` по аналогии с `pf_account_v1`), чтобы не ломать
инвариант «медаль = навсегда» и подсчёт «Легенды».

✅ **Сделано, с ОДНИМ отходом от плана:** бейджи `season_bronze`…`season_diamond` (`comp:true`,
кумулятивные — достиг Diamond, получил и все четыре ниже) выдаёт `ACH.onSeasonDivision()`;
косметика — `Leaderboard.season.claimReward(divisionIdx)`/`.reward()` в `pf_season_rewards_v1`,
как и спланировано. ⚠️ **Отход:** план предполагал подавать косметику «через существующую
систему скинов» — эта система **убрана целиком** между написанием плана и реализацией (см.
`CLAUDE.md`: один скин `neon`, `SKIN_DEFS` и выбор скина не существуют больше). Вместо неё —
акцентный цвет (кружок-индикатор рядом с дивизионом на вкладке «Сезон»), самодостаточный, без
зависимости от снятой системы. Мягкая валюта — не сделана (не было в MVP-скоупе).

### 5. Android: маппинг на нативный Play Games
