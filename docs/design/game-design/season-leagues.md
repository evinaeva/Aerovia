# PlaneFlow — Лига сезона (сезонные соревнования)

> **Статус: ПЛАН / предложение.** Ничего из этого пока не в коде. Документ описывает,
> как перенести в PlaneFlow механику **Google Play «Leagues»** (кейс Subway Surfers,
> [play.google.com/console/about/subway-surfers-leagues-casestudy](https://play.google.com/console/about/subway-surfers-leagues-casestudy/))
> — **ограниченное по времени соревнование на уже-считаемых метриках + дивизионы + награды.**
>
> Строится поверх уже готового соревновательного каркаса — см.
> [`leaderboards.md`](leaderboards.md) (каркас `Leaderboard`/`Account`, `periodBucket`,
> ранг-медали) и боевой бэкенд [`backend-firebase.md`](backend-firebase.md).
> Обзорно — в [`../../backlog.md`](../../backlog.md).

---

## ⚠️ Это НОВОЕ продуктовое решение (отход от «без сезонов»)

Дизайн-док рейтингов фиксирует прежний выбор овнера:
[«ранг-медали — пороговые **НАВСЕГДА**, без сезонов/перевыдачи»](leaderboards.md#ранг-медали--пороговые-навсегда).
**Сезонная лига этот выбор нарушает** — по определению это повторяющееся окно с перевыдачей.
Как и сами рейтинги когда-то были сознательным отходом от «лёгкой меты», лига — это
осознанный шаг, который овнер должен подтвердить **до** реализации (см. «Открытые вопросы»).

Чтобы конфликт был минимальным, план **разводит два класса наград** (см. ниже): постоянные
пороговые медали остаются как есть, а «сезонным» становится только косметика + дивизион-бейджи,
которые тоже пороговые-навсегда. То есть модель «навсегда» мы не ломаем — добавляем поверх неё
ротацию контента.

---

## Зачем (урок кейса)

Главный вывод кейса — **не награды, а цифры**: у Subway Surfers Leagues **+45 %
игрового времени** у записавшихся, **75 % записавшихся — уже вовлечённые игроки**,
**+3 % инсталлов** от самого ивента, и — ключевое для нас — успех достигнут
*«минимальными усилиями разработки, переиспользовав уже отслеживаемые метрики»*
(у них — монеты и время; у нас — **обслуженные борта / combo / streak / earned**).

Единственный настоящий рычаг ретеншена здесь — **повторяющийся повод вернуться**:
«до конца сезона 3 дня, ты 2-й в Серебряном дивизионе». Всё остальное — обвязка вокруг него.

## Дизайн

### 1. Сезон = новый бакет `periodBucket`

Сейчас `periodBucket` отдаёт `all` / `YYYY-MM` / `YYYY-Www`. Добавляем **`season`**:
детерминированное 2-недельное окно UTC от фиксированного якоря.

```
SEASON_EPOCH = <фикс. понедельник 00:00 UTC>   // константа в конфиге
SEASON_LEN   = 14 * 24 * 3600 * 1000
seasonIndex(ts) = Math.floor((ts - SEASON_EPOCH) / SEASON_LEN)   // 0,1,2,…
ключ бакета     = 'S' + seasonIndex             // напр. 'S37'
номер для показа = seasonIndex + 1              // «Сезон 38»
```

Свойства сохраняем те же, что у существующих периодов: чистая функция, UTC, одинаковая
на клиенте и будущем сервере, срез = фильтр журнала по текущему бакету (без разрушающего
сброса — старые счета живут в all-time). Длину/якорь — в конфиг, чтобы менять без правки логики.

### 2. Дивизионы (Bronze → Diamond)

Вместо одного глобального топа (где ты 4000-й) — **бракеты по силе**: мотивирует быть 2-м из 30.

- 5 дивизионов: 🥉 Bronze · 🥈 Silver · 🥇 Gold · 💠 Platinum · 💎 Diamond.
- **Назначение:** первый сезон игрока — по перцентилю его all-time `bestScore`; далее —
  **promotion/relegation** по итогу прошлого сезона (топ ~20 % ↑, низ ~20 % ↓).
- ⚠️ **Настоящие кросс-игроковые дивизионы без сервера невозможны** (та же причина, что
  глобальный топ — см. [`leaderboards.md`](leaderboards.md#статус-это-каркас-не-боевой-рейтинг)).
  На mock-провайдере дивизион — это честная **демонстрация** против сид-ботов
  ([`07-audio-services.ts` → `seedBots`](../../../src/game/07-audio-services.ts)): бракет и
  ботов подбираем под уровень игрока. Боевые дивизионы приходят с бэкендом (Фаза 3).

### 3. Награды — два класса (мирим с «навсегда»)

| Класс | Что | Модель | Легенда? |
| --- | --- | --- | --- |
| **Постоянные бейджи дивизиона** | «Достиг Gold», «Достиг Diamond» и т.п. | пороговые **навсегда** (как ранг-медали) — новый флаг `season`/`comp` | **нет** (мимо «Легенды», как `comp:true`) |
| **Сезонная косметика** | 1 скин/цвет-акцент за сезон (или за место) | **ротация**, хранится в отдельном реестре наград, не в `unlocked`-Set | н/д |
| (опц.) Мягкая валюта | бонус `money` за место | разовое начисление | н/д |

Ключевой приём: **`ACH.unlocked`-Set не трогаем ротацией** — туда попадают только
пороговые-навсегда бейджи дивизиона. Ротирующиеся призы (скин сезона) идут в **отдельный
стор наград** (`pf_season_rewards_v1` по аналогии с `pf_account_v1`), чтобы не ломать
инвариант «медаль = навсегда» и подсчёт «Легенды». Косметику подаём через существующую
систему скинов.

### 4. UX — отсчёт и «запись»

- **Отсчёт** «До конца сезона: N дн.» — на экране рейтинга (рядом с `lbPeriod`) и чипом
  на старт-экране (`updateStartChips`). Это и есть рычаг +45 %.
- **Приглашение в лигу — сначала своим активным** (у SYBO 75 % записавшихся уже играли):
  тост «Стартовал Сезон N» показываем в первую очередь тем, у кого уже есть `bestScore`.
  Пуш — через существующий канал доставки (PWA SW / APK).
- **Шеринг-карточка** «Я — #X в Серебряном дивизионе, Сезон N» → у них ивент дал +3 %
  инсталлов; для нас это бесплатный маркетинг (Web Share API / нативный шер).

### 5. Android: маппинг на нативный Play Games

Play Games даёт **временны́е таблицы (daily/weekly/all-time) из коробки** → «текущий сезон»
можно отразить на weekly/период-таблицу Play Games и получить **реальные кросс-игроковые
позиции бесплатно**, без своего сервера. Дивизионы/промо остаются клиентскими (или на
бэкенде). Проверить точные time-span Play Games при реализации.

## Фазы

- **Фаза 0 — решение (блокер).** Овнер подтверждает сезоны (отход от «без сезонов»),
  выбирает длину сезона, число дивизионов и тип наград. Без этого не начинаем.
- **Фаза 1 — MVP на mock, без бэкенда (низкая–средняя).**
  `season`-бакет в `periodBucket` (+тесты) → вкладка «Лига сезона» на экране рейтинга с
  отсчётом → клиентские дивизионы против сид-ботов → постоянные бейджи дивизиона (мимо
  «Легенды») → 1 сезонный косметический приз в отдельном сторе. Честная демка — ровно как
  текущий mock-рейтинг.
- **Фаза 2 — Android-нативка (средняя).** Отразить текущий сезон на период-таблицу Play
  Games (реальные позиции) → приглашение/возврат пушем; шеринг-карточка.
- **Фаза 3 — боевые дивизионы и награды (высокая, блок на бэкенд).** Требует Firebase
  ([`backend-firebase.md`](backend-firebase.md)): сервер считает дивизионы, промо/релегейт,
  выдаёт награды, серверный анти-чит. До него всё соревновательное — демонстрационное.

## Что НЕ делаем (границы)

- **Реальные Play Points из кейса.** Спонсорская программа Google — партнёрка для крупных
  студий (SYBO), инди туда сейчас не зайти. Награды держим **внутриигровыми** (косметика/бейджи).
- **Сложный матчмейкинг** (возраст, 33 страны, античит-подбор) — не нужен; хватит бракетов по `bestScore`.
- **IAP-first награды** — игра пока без покупок; призы косметические. (Когда появятся IAP —
  сезоны это готовый крючок; см. монетизацию в [`../../backlog.md`](../../backlog.md).)
- **Ломать «медаль = навсегда»** — ротирует только косметика в отдельном сторе.

## Анти-чит и честность

Как и у базового рейтинга ([`leaderboards.md`](leaderboards.md#анти-чит-важно-для-боевого-бэкенда)):
клиентский счёт подделывается тривиально, отправка изолирована в `submitRun` (одна точка).
На mock сезонные награды — **демонстрационные**; боевая целостность (реальные дивизионы,
призы за деньги/место) требует серверной валидации из Фазы 3.

1. **Сезоны — да?** Это отход от «ранг-медали навсегда, без сезонов». Подтверждаем?
2. **Длина сезона** — 2 недели (как у SYBO) или другое?
3. **Метрика лиги** — Survival `served`, или отдельный сезонный скоринг (combo/streak/earned)?
4. **Награды** — только косметика+бейджи, или ещё `money`-бонус? Сколько дивизионов (5?)?
5. **Порядок** — делать Фазу 1 (MVP на mock) сейчас, или ждать бэкенд и делать сразу боевую?
