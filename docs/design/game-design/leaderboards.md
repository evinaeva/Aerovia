> Дизайн-документ о **каркасе** соревновательного слоя: глобальные таблицы лидеров
> (за всё время / месяц / неделю) для режима **Survival** и привязанные к ним медали.
> Введено решением продукт-овнера **2026-06-14** (см. примечание в
> [`meta_progression.md`](meta_progression.md#вне-скоупа-отложено--не-строить-без-нового-решения) —

> `ACH.onRank`, флаг `survival`/`currentMode()`). Тесты: [`tests/leaderboard.test.mjs`](../../../tests/leaderboard.test.mjs).

## Статус: это КАРКАС, не боевой рейтинг

Игра — статичный PWA на GitHub Pages, **без сервера**. Глобальную таблицу (чужие
результаты по всем игрокам) без бэкенда сделать нельзя. Поэтому сделан **каркас**: вся
клиентская обвязка (приём счёта, периодизация, таблица, медали) работает уже сейчас на
**локальном mock-провайдере**, а реальный сервер подключается **одной заменой провайдера**,
не трогая игру. Это тот же приём, что уже применён для аналитики (`Analytics.sink`).

Что РАБОТАЕТ сейчас (на mock):
- режим Survival как шов (флаг + старт + отправка счёта на финале захода);
- срезы all-time / month / week из единого журнала счётов;
- экран рейтинга (кнопка-трофей в главном меню) с демо-ботами;
- пороговые ранг-медали (топ-100 / топ-10 / №1), выдаются навсегда.

Что ЕЩЁ НЕ сделано (→ [`backlog.md`](../../backlog.md)): реальный бэкенд, настоящие
аккаунты (Google Play Games / OAuth), серверный анти-чит, обновление privacy/Data Safety,
геймплейный баланс самого Survival, видимая кнопка запуска Survival.

## Архитектура — свопаемый провайдер

```
игра (endLevel, экран рейтинга)
        │  submitRun() / top() / me() / bestScore()
        ▼
   Leaderboard  ──── provider ────►  [ mockProvider ]   ← сейчас (localStorage + боты)
        │                            [ свой API/Firebase/Supabase ]  ← потом, одной заменой
        └── account ──► Account ──── authProvider ──►  [ mockAuth ]  ← сейчас (локальный аккаунт)
                                                       [ Google Play Games / OAuth ] ← потом
```

- **`Leaderboard.provider`** — объект `{submit(entry), top(period, mode)}`. По умолчанию
  локальный mock. Реальный бэкенд реализует тот же интерфейс через сеть — `submitRun()`/`top()`
  в игре не меняются. (Геттер/сеттер с проверкой формы, как `Analytics.sink`.)
- **`Account.authProvider`** — `() => Promise<{id, name, provider}>`. Сейчас mock «локальный
  аккаунт». Реальный вход подменяет только эту функцию.
- Дев-доступ из консоли: `window.PFLeaderboard` (`.top('week')`, `.bestScore('survival')`,
  `.account.current()`).

**Почему так (CLAUDE.md):** рейтинг — это *логика*, она одна для всех скинов/биомов/карт.
Скин — лишь поле `skin` в записи счёта; никаких `if(SKIN==='…')`.

## Метрика и ранжирование

- **Метрика Survival = число обслуженных бортов (`served`) за заход.** На финале захода
  (`endLevel`, когда `currentMode()==='survival'`) счёт уходит в `Leaderboard.submitRun`.
- **Лучший-на-аккаунт:** в таблице у игрока одна строка — его **лучший** заход в данном
  срезе. Журнал хранит все отправки; ранжирование берёт максимум на аккаунт.
- Сортировка: счёт ↓, при равенстве — кто раньше поставил рекорд.

## Периоды — `periodBucket(ts)`

Чистая детерминированная функция (UTC), одинаковая на клиенте и будущем сервере:

| Период | Ключ | Как считается |
| --- | --- | --- |
| `alltime` | `'all'` | весь журнал |
| `month` | `'YYYY-MM'` | календарный месяц UTC |
| `week` | `'YYYY-Www'` | неделя по **ISO-8601** (год определяется четвергом недели) |

Срезы — это **скользящие окна, без разрушающего сброса**: «за неделю/месяц» = фильтр того же
журнала по текущему бакету. Когда наступает новая неделя/месяц — таблица естественно
обнуляется (старые счета остаются в all-time). Сервер обязан считать периоды так же.

## Идентичность — реальные аккаунты

Выбор продукт-овнера — **реальные аккаунты** (общий профиль между устройствами, крепче
анти-чит). В каркасе это эмулируется: `mockAuth` отдаёт «локальный аккаунт» с устойчивым id
(из `pf_uid`) и ником, который игрок задаёт сам. Реальный вход (Google Play Games для
Android-сборки, OAuth для веба) встаёт в `Account.authProvider`, не меняя остальной код.

- Аккаунт хранится в своём ключе `pf_account_v1` и **переживает «Сбросить прогресс»** (как
  язык и медали) — личность не привязана к прогрессу кампании.
- ⚠️ Настоящие аккаунты с никами = персональные данные → при подключении реального входа
  обновить [`privacy.html`](../../../privacy.html) и форму **Play Data Safety**
  ([`play-data-safety.md`](../../play-data-safety.md)). Сейчас это ещё не требуется (всё локально).

## Ранг-медали — пороговые НАВСЕГДА

Выбор продукт-овнера — **только пороговые навсегда** (без сезонов/перевыдачи). Достиг
порога хоть раз в любом срезе — медаль твоя:

| Медаль | Условие |
| --- | --- |
| 🏅 **На табло** (`rank_top100`) | Попасть в мировой топ-100 (любой период). |
| 🏆 **Десятка** (`rank_top10`) | Попасть в мировой топ-10 (любой период). |
| 👑 **Номер один** (`rank_1`) | Стать №1 в глобальном рейтинге. |

- Выдаёт `ACH.onRank(ranks)` сразу после `submitRun` (берёт лучший достигнутый ранг по
  всем срезам). Идёт **мимо `RUN_CAP`** (это редкое крупное событие, не шумная накопиловка).
- Помечены **`comp:true`** → **НЕ входят в требование «Легенды»**. Легенда = мастерство
  соло-игры, не должна зависеть от чужих результатов и наличия сервера.
- ⚠️ На mock-провайдере ботов мало (8), поэтому «топ-10» достигается легко — это артефакт
  заглушки; на реальном бэкенде пороги обретают вес.

## Survival — шов под будущий режим

- `survival` (флаг) + `currentMode()` — единый источник истины о режиме (заменил инлайновые
  тернарники). `startSurvival()` стартует заход; на финале `endLevel` шлёт счёт в рейтинг.
- ⚠️ **Баланс/кривая сложности и условие смерти Survival — НЕ в этом каркасе.** Сейчас режим
  переиспользует стартовую раскладку, чтобы каркас рейтинга/медалей был играбелен и тестируем.
  Доведение режима (бесконечный спавн, рост темпа, экран итогов) — отдельный геймплейный
  проход; до него видимую кнопку «Survival» в меню не выводим (косметика — соседний трек).

## Анти-чит (важно для боевого бэкенда)

Клиентский счёт подделывается тривиально. Каркас изолирует отправку в `submitRun` (одна
точка), поэтому при появлении сервера можно добавить серверную валидацию (разумные потолки
темпа/счёта, подпись/токен сессии, отбраковку выбросов) **не переписывая игру**. До тех пор
рейтинг — демонстрационный.
