# PlaneFlow — Условия на звёзды: разбор референса и наша модель

> Каталог **возможных сочетаний условий 1★/2★/3★**, собранный из референс-игры
> **«Airport Control»** (48 уровней, 53 скриншота в [`star-conditions-reference/`](star-conditions-reference/)),
> плюс то, как мы переносим это в нашу модель уровня (`Objective`) и в **валидатор
> проходимости**. Формула сложности и кривая — в [`difficulty_curve.md`](difficulty_curve.md);
> схема конфига уровня — в [`level-pattern.md`](level-pattern.md); экономика — в
> [`economy.md`](economy.md).

Скриншоты — это «словарь жанра»: какими бывают цели и из чего складываются пороги на
звёзды у похожей игры. Мы не копируем числа уровней, мы берём **набор условий** и
**паттерны их комбинирования**, чтобы (а) дать их настраивать в редакторе и (б) научить
валидатор проверять достижимость каждого тира.

---

## 1. Что вообще бывает условием звезды (метрики)

Из 48 уровней референса каждый тир (1★/2★/3★) — это **И-связка** (AND) одного или
нескольких условий из этого набора:

| Метрика | Что значит | Иконка в референсе | Наше поле `Objective` |
| --- | --- | --- | --- |
| **Обслужено бортов** | принять/выпустить ≥ N бортов | синий самолёт | `stars: number[]` (есть) |
| **Заработок** | накопить ≥ M денег к концу | монета / зелёный «нал» | `money?: number[]` (новое) |
| **Осталось жизней** | сохранить ≥ L жизней | сердце | `lives?: number[]` (новое) |
| **Уложиться по времени** | закрыть цель за ≤ T (быстрее = больше ★) | часы в строке тира | `timeTier?: number[]` (новое) |
| **Дожить до времени** | продержаться ≥ T (survival) | красные часы | `survive?: number[]` (новое) |
| **Сделать апгрейдов** | прокачать/открыть боксы ≥ U | зелёная станция | `upg?: number[]` (есть) / `metric:'upgrades'` |
| **Без просрочек** | не больше K наземных просрочек | — (у нас) | `maxLate?: number[]` (новое, обобщает 2★) |
| **Без крушений** | не больше K крушений | — (у нас) | `maxCrash?: number[]` (новое, обобщает 3★) |

Плюс **конфиг уровня** (не условие звезды, а то, что делает карту сложнее): штрафы
late/crash (30–100%), число жизней на старте (1/2/3), спецборты (VIP), волны/час-пик,
марафон (поток растёт со временем), погода. Всё это у нас уже есть ручками уровня
(`latePenalty`, `crashPenalty`, `events`, `pace`, `weather`…).

## 2. Паттерны комбинирования тиров (главное)

Тиры почти всегда **монотонны по сложности** (3★ строго труднее 2★ труднее 1★). Способы,
которыми референс «утяжеляет» верхний тир, повторяются — это и есть пресеты для
авто-генератора и инварианты для валидатора:

1. **Чистая лестница по числу** — 1★/2★/3★ отличаются только порогом бортов.
   _L1 6/7/8 · L28 70/75/80 · L34 70/80/100 · L43 60/80/100 · L47 130/165/200._
2. **Счёт + тайм-гейт на верхушке** — 1★/2★ только число, 3★ добавляет «за время T».
   _L7 24@5:00 · L18 50@8:00 · L23 50@8:20 · L30 70@11:00 · L37 80@10:00 · L41 70@10:30._
3. **Одинаковый счёт, разное время** — 2★ и 1★ требуют то же N, отличаются временем.
   _L8 20@4:10 / 20@5:00 / 16 · L10 20 / 20@5:00 / 20 · L24 50@8:30 / 50@10:00 / 35._
4. **Счёт + деньги** — каждый тир = борта + порог заработка.
   _L9 24+25k / 21+10k / 18+5k · L19 40+30k / 35+20k / 25+10k · L22 50+50k / 44+30k / 34+10k._
5. **Счёт + жизни** — верхние тиры требуют сохранить больше жизней.
   _L14 3♥+32 / 2♥+28 / 25._
6. **Счёт + апгрейды** — борта плюс число прокачек.
   _L16 42+18up / 36+18up / 30+12up._ Либо метрика — сами апгрейды: _L11 14/16/18 up._
7. **Время + деньги (фикс. деньги)** — все тиры требуют те же деньги, отличаются временем
   (быстрее накопить = больше ★). _L32 36$ @21/18/15 мин · L35/38/40/42/44 аналогично._
8. **Чистый survival по времени** — дожить до T; верхний тир может ещё требовать число.
   _L45 8/9/10 мин · L48 10:00/12:30/15:00 · L46 8:00 / 10:00 / 90+12:00 · L27 30@7 / 45@8 / 60@10._

**Инверсия осей** (важно для UI/валидатора): на «гонках» лимит времени стоит сверху как
жёсткое окно, а целью становится «No limit» по числу — то есть **либо** ограничено время и
свободно число (race/survival), **либо** ограничено число и свободно (или дополнительно
ограничено) время. Эти два режима у нас = `race` (свободное число за `time`) против обычной
цели `served` с опциональными `timeTier`.

## 4. Как это ложится на нашу модель `Objective`

Сейчас (`src/game/04-config-levels.ts`, `_contracts.d.ts`):

```ts
interface Objective {
  metric: 'served' | 'upgrades';
  stars: number[];   // [1★,2★,3★] пороги по метрике
  target?: number;   // = stars[2]
  time?: number;     // жёсткий лимит времени (сек)
  race?: boolean;    // «сколько успеешь за time»
  upg?: number[];    // пер-тир пороги апгрейдов (с metric:'served')
}
```

Звёзды считаются в `computeStars()` (`08-gameplay.ts`): **1★** = цель, **2★** = +без
наземных просрочек, **3★** = +без крушений. То есть «без просрочек/крушений» сейчас
**зашиты** в тиры 2/3, а число — это только `stars[]`.

**Расширение (этот PR):** «без просрочек/крушений» становится настраиваемым
(`maxLate`/`maxCrash` — пер-тир), и добавляются недостающие из референса метрики. Все новые
поля — **опциональные пер-тир массивы длины 3**, выровненные на `[1★,2★,3★]`; тир
засчитывается, если выполнены **ВСЕ заданные** условия этого тира (AND):

```ts
interface Objective {
  metric: 'served' | 'upgrades' | 'survival';
  stars: number[];        // борта / апгрейды / секунды (для survival)
  target?: number;
  time?: number; race?: boolean;
  upg?: number[];         // апгрейды как доп-условие
  money?: number[];       // ≥ заработок (банк) к концу
  lives?: number[];       // ≥ осталось жизней
  timeTier?: number[];    // ≤ закрыть за столько секунд (быстрее = выше ★)
  survive?: number[];     // ≥ продержаться столько секунд (survival)
  maxLate?: number[];     // ≤ наземных просрочек (умолч. наследует старое 2★)
  maxCrash?: number[];    // ≤ крушений (умолч. наследует старое 3★)
}
```

Соответствие паттернов §2 нашим полям:

| Паттерн §2 | Поля |
| --- | --- |
| 1. лестница по числу | `stars` |
| 2. счёт + тайм-гейт сверху | `stars` + `timeTier` (на верхних тирах) |
| 3. тот же счёт, разное время | `stars` (плато) + `timeTier` |
| 4. счёт + деньги | `stars` + `money` |
| 5. счёт + жизни | `stars` + `lives` |
| 6. счёт + апгрейды | `stars` + `upg` |
| 7. время + фикс. деньги | `metric:'served'`+`money` (плато) + `timeTier` |
| 8. survival по времени | `metric:'survival'`, `stars`=секунды (или `survive`) |

**Обратная совместимость:** все старые уровни валидны без изменений — новые поля
опциональны. Важно: в текущем коде `computeStars()` звёзды считаются **только по числу**
(порог `stars[]` + опц. `upg[]`) — гейта «без просрочек/крушений» в игре пока нет (это
замысел из QA-дока, но не реализация). Поэтому новые `maxLate`/`maxCrash` — чисто
**opt-in**: если не заданы, гейта нет и поведение байт-в-байт прежнее. Уровень может
включить их там, где хочет качественный порог на 2★/3★.

## 5. Рубрика валидатора проходимости (1★/2★/3★)

`validatePassable(lv)` проверяет, что **каждый** тир в принципе достижим компетентным
игроком. Источник оценок — задокументированная модель (`paceInterval`, `paceCap`,
`AIR_BASE`, `levelEconomy`, `levelFlow`). Тир «непроходим», если нарушено хоть одно:

1. **Монотонность.** Для «≥»-условий (`stars`,`money`,`lives`,`upg`,`survive`) пороги не
   убывают по тирам; для «≤»-условий (`timeTier`,`maxLate`,`maxCrash`) — не возрастают.
   Нарушение = ошибка (тир логически недостижим/перепутан).
2. **Пропускная способность.** Максимум обслуживаемых за смену
   `servableMax ≈ throughput(pace) × окно_времени` должен покрывать `stars[i]`.
   `throughput` ограничен интервалом спавна и `paceCap`; окно = `time` (или щедрая оценка
   для «No limit»). Если `stars[i] > servableMax` → 1★/2★/3★ недостижимо.
3. **Время на тир.** `timeTier[i]` ≥ минимально нужного, чтобы обслужить `stars[i]` бортов
   на данном темпе. Иначе тир требует «быстрее физически возможного».
4. **Деньги.** `money[i]` ≤ реалистичного заработка
   `startMoney + svcReward × avgNSvc × stars[i] × skillMult` (см. `levelEconomy`). Иначе
   порог денег недостижим.
5. **Жизни.** `lives[i] ≤ START_LIVES`. И при «одна ошибка = конец» (1♥) тир,
   требующий 0 просрочек/крушений, помечается как «хардкорный», но не блокируется.
6. **Деньги не блокируют звезду.** «Набор» (открыть/прокачать боксы, открыть направления
   ВПП) по карману competent-игроку — это уже гарантирует `levelEconomy` (generosity ≥ 1);
   валидатор лишь подсвечивает, если `kitCost × generosity` не покрывается потоком.
7. **Хотя бы одна ВПП-направление открыто** и **есть чем обслуживать** (открытые боксы или
   деньги на открытие до набора цели) — иначе старт нерабочий.

Вывод валидатора — список `{ tier: 1|2|3, ok: boolean, reasons: string[] }`; редактор
показывает 🟢/🔴 по каждому тиру с причинами. Авто-генератор (§6) гоняет уровень через эту
рубрику и ослабляет пороги, пока все три тира не станут проходимыми.

## 6. Авто-генератор уровня (сложность → ВСЕ факторы)

`autoDifficulty(target, lv?, opts?)` (`target` 0..1) — **layout-aware генератор**: читает
разметку и ручные настройки оператора из `lv` и выдаёт согласованный набор **всех**
факторов победы и экономики. Ключевой принцип: **все факторы активны всегда** — сложность
живёт в их *параметрах и сочетании*, а не в наборе включённых условий. Два уровня с одинаковым
числом бортов могут требовать на 3★ либо высокую кассу при нуле аварий, либо мелкую сумму,
которую набьёшь и с потерями.

- **События и погоду НЕ генерирует** — это ручки оператора (по требованию).
- **Темп `pace`** = `target` напрямую (главная ось).
- **Пороги бортов `stars`** — из ёмкости карты: открытые ангары + открытые направления ВПП +
  полная прокачка (`1 + maxUp·K.UP_SPEED`, т.е. +25%/уровень).
- **Все условия длиной 3** (`timeTier`/`money`/`lives`/`maxLate`/`maxCrash`/`upg`),
  масштабируются от щедрых к жёстким. `money` привязаны к реально зарабатываемому (тот же
  `levelEconomy`, что в валидаторе) и опускаются, если экономика их не тянет.
- **Цены единые на карту** (`openCost`/`upgCost`/`rwOpenCost`), `maxUp`/`minUp`, `startMoney`,
  штрафы — тоже генерируются.
- **`opts.locked`** — поля, заданные оператором: их НЕ перезаписываем (читаем как вход).
- **`opts.archetype`** — *акцент*: вектор «строгости» групп (деньги/время/качество/апгрейды/
  трафик) при том же `target` — даёт разнообразие без роста raw-сложности. См. `ARCHETYPES`.

Результат прогоняется через `validatePassable`; пороги ослабляются до зелёного на 1/2/3★.

### 6.1 Кривая кампании и капстоны

`campaignTarget(index, curveOpts?)` маппит **номер уровня → target** насыщающейся кривой:
обучение (пологий низ) → подъём (ease-out) → **плато < 1.0**; каждый `capstoneEvery`-й уровень —
**капстон** на ~1.0. Расширение кампании (+50 уровней) = больше уровней на том же плато:
raw-сложность не растёт выше «предела компетентного игрока», разнообразие даёт **ротация
акцентов** (`archetypeForIndex`). Ручки кривой — `K.CURVE`
(`tutorialLen`/`rampEnd`/`plateauHeight`/`capstoneEvery`), редактируются в `tuning.html`
(вкладка «Сложность»). Per-level override — `Level.target` / `Level.archetype`.
