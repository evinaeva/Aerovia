# Бюджет памяти и требования к PR (Android 17)

> Первоисточник: Google, «Prioritizing memory efficiency: steps for Android 17»
> (Android Developers Blog, июнь 2026). Этот документ переносит его требования на
> **наш** контекст и фиксирует правила, которые должны соблюдаться **в каждом PR**.

## Почему это важно

Android 17 вводит **жёсткий лимит RAM на процесс**, зависящий от объёма памяти
устройства. Если приложение его превышает, система **убивает процесс молча, без
стектрейса** (в `ApplicationExitInfo`: причина `REASON_OTHER`, описание содержит
`MemoryLimiter:AnonSwap`). Мотив Google: одно текущее приложение с раздутой памятью
(особенно с foreground-сервисом, защищённым от Low Memory Killer) заставляет систему
убивать десятки аккуратных фоновых приложений и рушить их состояние. Итог —
приложения-«обжоры» будут первыми под нож.

## Ключевой нюанс: мы — WebView-игра, а не нативное приложение

Aerovia/PlaneFlow — это веб-игра (`src/`), которую Capacitor упаковывает в тонкую
нативную оболочку (`webDir: www/`, см. [`capacitor.config.ts`](../capacitor.config.ts)).
Поэтому:

- **«Процесс», который считает лимит, — это WebView.** Наша память — это НЕ Java-heap,
  а: **декодированные битмапы** спрайтов, **canvas backing store**, **JS-heap** и
  аудио-графы Web Audio.
- Половина советов Google ложится **косвенно** — таблица ниже переводит их на наш стек.

| Совет Google (нативка) | Наш эквивалент (WebView) |
| --- | --- |
| Лимит RAM, молчаливый kill | Держать декодированный набор битмапов + JS-heap малым, особенно в фоне |
| R8 / shrink / ProGuard | Применяется к генерируемой нативной оболочке (`android/`, см. `scripts/setup-android.mjs`) — код игры это JS в WebView |
| `onTrimMemory(UI_HIDDEN)` | `visibilitychange` → `hidden` (уже: пауза rAF, [`10-scene-loop.ts:193`](../src/game/10-scene-loop.ts)) |
| `onTrimMemory(BACKGROUND)` | Событие `freeze` (Page Lifecycle API) → сбросить регенерируемые кэши |
| Downsample / `inSampleSize` | Исходный размер PNG ≤ макс. экранного при `dpr≤2` |
| `RGB_565` вместо `ARGB_8888` | Где нет прозрачности (фон, апрон) — хранить/декодировать без альфы |
| Приоритет векторов | SVG-символы вместо растра, где glow не запечён |
| Дедуп битмапов (⚠️ в heap dump) | Один файл на визуально одинаковый арт; варианты — рантайм-тинтом |
| `bitmap.recycle()` | Отпускать крупные canvas сразу; чистить кэши на `hidden`/`freeze` |
| LeakCanary / heap dump | Никаких неограниченных `Map`-кэшей, снятие слушателей, отсутствие пер-кадровых аллокаций |
| `ProfilingManager` OOM/Anomaly | (Опц.) нативный хук: читать `ApplicationExitInfo` на старте → аналитика |

## Требования к КАЖДОМУ PR

Это чек-лист, который обязан выполняться в любом изменении. CI-гард
([`scripts/check-asset-budget.mjs`](../scripts/check-asset-budget.mjs), шаг в
[`deploy.yml`](../.github/workflows/deploy.yml) — Фаза 4) проверяет пункты, которые
можно проверить машинно: байт-бюджет `www/assets`, размер битмапов, дубликаты по хэшу.

### Ассеты
- [ ] Новый битмап не добавляется, если то же самое можно нарисовать **SVG/процедурно**
      или получить **рантайм-тинтом** существующего спрайта.
- [ ] **Никаких дубликатов.** Файлы с одинаковым содержимым (одинаковый хэш) недопустимы —
      хранить один файл, варианты делать тинтом/трансформом.
- [ ] Исходный размер растра **≤ максимального экранного** при `dpr = 2`. Никакого
      2×-переразмеренного арта. Где прозрачность не нужна — сохранять без альфа-канала.
- [ ] Dev/референс-ассеты (`*_src_reference.png` и т.п.) и наборы **только для
      tuning-воркбенча** (`assets/skins/**`) **не попадают в `www/`** (то есть в APK/OTA/PWA).
- [ ] Суммарный размер `www/assets` не превышает бюджет (см. ниже). Рост бюджета —
      только с обоснованием в описании PR.

### Рантайм-память
- [ ] **Нет неограниченных кэшей.** Любой `Map`/объект, кэширующий изображения или canvas,
      имеет границу вытеснения (LRU/лимит) **или** очищается на `hidden`/`freeze`.
- [ ] При уходе в фон регенерируемая память освобождается; rAF-цикл **остаётся
      остановленным** в фоне (не регрессировать поведение из `10-scene-loop.ts`).
- [ ] Крупные временные canvas (share-карта 1080×1080, offscreen-буферы) — **локальные**
      и отпускаются сразу после использования; не кэшируются на глобалах без границы.
- [ ] Сохраняется кап `dpr = min(devicePixelRatio, 2)` ([`06-state-layout.ts:70`](../src/game/06-state-layout.ts)).
      Ни один canvas не аллоцируется больше, чем вьюпорт × dpr.
- [ ] **Нет пер-кадровых аллокаций** в render/step-цикле (не создавать массивы/объекты/строки
      каждый кадр — переиспользовать буферы).

### Нативная сборка
- [ ] Release-сборка Android держит R8-shrink + resource-shrink (не регрессировать
      настройки в `scripts/setup-android.mjs`).
- [ ] Новая npm-зависимость добавляется только со взвешиванием её веса в бандле.

### Проверка перед пушем
- [ ] Загрузить игру → свернуть (скрыть вкладку) → RAM падает, rAF остановлен →
      развернуть → корректно возобновляется.
- [ ] JS-heap не растёт безгранично при повторных переигровках уровня.

## Бюджеты (конкретные числа)

| Метрика | Потолок (CI-гейт) | Цель | Где мерить |
| --- | --- | --- | --- |
| `www/assets` суммарно | **≤ 19 МБ** (не расти от текущего) | **≤ 12 МБ** после Фазы 2 (даунсемплинг) | CI-гард по размеру |
| Один битмап (исходный файл) | **≤ 2.3 МБ** на диске | ≤ 1.5 МБ | CI-гард |
| Пиксельный размер битмапа | **≤ 2048 px** по большей стороне | — | CI-гард (по возможности) |
| Дубликаты по хэшу в `www/assets` | **0** | 0 | CI-гард |

Замер после доп. чистки (2026-07-02): `www/assets` ≈ **10.94 МБ** (Фаза 0 свела ~28 →
~18 МБ чисткой skins/dupes; Фаза 4 добила до ~15.6 МБ, убрав дубль/dev-мусор; доп.
чистка мёртвых `sprite_hangar.png`/`sprite_gate.png`/`sprite_vpp_conn.png` — до 13.76 МБ;
консолидация иконок услуг `svc_*_bot`/`svc_*_side` → общий файл на услугу, 9→3 — до
10.94 МБ). **Уже ниже цели Фазы 2 (≤12 МБ) — без единого пересжатия существующего
арта**, только удалением мёртвого/дублирующегося. CI-гард
стоит на потолке 19 МБ (запас под текущим — намеренный; потолок опускаем отдельным
решением, чтобы не блокировать несвязанные PR случайным ростом).
«Потолок» — это то, что CI не даёт превысить (защита от регресса); «Цель» — куда
двигаемся Фазой 2. Главное правило: **потолок только опускается**, любое увеличение
требует явного обоснования в описании PR.

**Фаза 2 — правильный размер арта — числа посчитаны (2026-07-01), сам ресайз PNG и
визуальная проверка — вручную:**

Часть спрайтов масштабируется игровой единицей `ui` ([`06-state-layout.ts`](../src/game/06-state-layout.ts)),
которая жёстко зажата в `[0.7, 1.5]` **независимо от размера экрана** — у них есть
точный физический потолок, который можно посчитать и подрезать под него. У фоновых
спрайтов на весь вьюпорт (`sprite_back_full.png`, `sprite_apron.png`, `sprite_hud.png`,
`sprite_vpp.png`) такого потолка нет — они растягиваются на `W×H` устройства (или на
длину ВПП), а `W`/`H` — это сырые `window.innerWidth/innerHeight`, ничем не
ограниченные сверху (канвас — `position:absolute;inset:0`, без `max-width`). При
потолке `dpr=2` эти исходники уже сейчас **не хватает** на части крупных ландшафтных
экранов (напр. флагман-телефон landscape CSS ≈915×412 × dpr2 ≈ 1830×824 — шире, чем
1672×941 у `sprite_back_full.png`) — то есть местами они, наоборот, **растягиваются
вверх**, а не вниз. Уменьшать их сейчас значит гарантированно ухудшить чёткость на
части устройств ради экономии, которую ещё нужно явно выбрать («до какого макс. экрана
держим 1:1 чёткость» — продуктовое решение, не инженерное); поэтому ниже только
UI-масштабируемые спрайты, для которых понижение размера строго безопасно.

Потолок физического пикселя для бокса: `PLANE_LEN(ui)=31·K.PLANE_SCALE·ui`, сторона
бокса `bw=min(PLANE_LEN·K.HANGAR_RATIO, апрон/2.4)` — второй член не связывает при
`ui=1.5` на реальных экранах (апрон растёт с `W` быстрее). Для борта:
`s=ui·0.5·K.PLANE_SCALE·vs`, `vs∈[K.PLANE_GND_SCALE 1.35, K.PLANE_SKY_SCALE 1.75]`,
рисуется `62·s`. Ни `HANGAR_RATIO`, ни `PLANE_SCALE`, ни `PLANE_SKY_SCALE` не
переопределяются per-level ([`04-config-levels.ts`](../src/game/04-config-levels.ts)) —
общие константы, потолок один на все уровни/карты.

| Файл(ы) | Сейчас | Физ. максимум на экране (`ui=1.5`, `dpr=2`) | Рекомендуемая цель | Экономия декод. памяти |

### Консолидация иконок услуг: 9 файлов → 3 (2026-07-02, код готов)

`svc_*.png` существовали в трёх предрисованных обрезках на услугу — `svc_fuel.png`
(верхние боксы), `svc_fuel_bot.png` (нижние), `svc_fuel_side.png` (боковые/конструктор),
и так же для `repair`/`board`. Визуальная проверка файлов показала: это **не повороты**
одной иконки (пиктограмма everywhere нарисована прямо, не боком/вверх ногами) — разница
только в том, где иконка сидит внутри холста (отступы), чтобы визуально прижиматься к
задней стене бокса, а та стена — сверху/снизу/сбоку в зависимости от ориентации.

Убрали `_bot`/`_side` (6 файлов), оставили один файл на услугу; позиционирование к
задней стене теперь считает код в [`09b-render-entities.ts`](../src/game/09b-render-entities.ts)
(`drawBay`) — якорь иконки поворачивается тем же `ang`, что и `sprite_hangar_base.png`
(так он оказывается у правильной стены для любой стороны бокса), а сама иконка рисуется
уже БЕЗ поворота (обратный `ctx.rotate(-ang)` перед `drawImage`), чтобы читаться прямо.
Пузырёк «нужды» над бортом и шаг в карточке борта уже использовали только базовый
вариант — там менять было нечего.

`SVC_ICON_SCALE`/`SVC_ICON_INSET` (в начале `drawBay`) — первая прикидка размера/отступа
на глаз по пропорциям исходных PNG, не точный ре-инжиниринг старых обрезков. Typecheck +
183 юнит-теста зелёные, рендер-геометрия по всем 4 сторонам проверена аналитически
(поворот переносит якорь к нужной стене для top/bottom/left/right — см. коммит), но
**пиксельный скриншот снять не удалось** — headless-превью держит вкладку в
`visibilityState:'hidden'`, из-за чего игровой rAF-цикл (намеренно, см. Фаза 1) не
рисует кадр даже после подмены `document.hidden`/`visibilitychange` из JS. Нужна
быстрая визуальная проверка на реальном устройстве/браузере после ресайза — заодно
можно подправить `SVC_ICON_SCALE`/`INSET`, если размер/отступ иконки покажется не тем.

### `sprite_back_full.png` / `sprite_apron.png` — числа разобраны отдельно (2026-07-02)

Оба растягиваются на весь `W×H` устройства без потолка, но по-разному (фон — на весь
экран, апрон — на его зону `≈0.495W×0.63H`, см. `layout()` в
[`06-state-layout.ts`](../src/game/06-state-layout.ts)), поэтому и вывод разный:

- **`sprite_back_full.png` (1672×941, RGBA) — НЕ уменьшать; на самом деле УВЕЛИЧИТЬ.**
  На рядовом Android-флагмане в ландшафте (CSS ≈915×412, `dpr=2`) нужно **≈1830×824
  физических px** — уже ШИРЕ нынешних 1672. Файл не «на пределе», а уже недотягивает
  до обычного телефона (на планшетах/desktop-PWA разрыв больше). Ориентир «любой
  телефон в ландшафте с запасом» — CSS ≈1000×500 при `dpr=2` → физический максимум
  **2000×1000**. **Рекомендуемая цель: 2000×1000** (или ровно **1920×1080** — стандартное
  разрешение, ~4% ýже расчётного, но удобное для тулинга и почти неотличимо на глаз).
  У него сейчас есть альфа-канал (`colorType=6` в PNG-заголовке), хотя по коду
  ([`09-render.ts:401`](../src/game/09-render.ts)) это непрозрачный нижний слой — когда
  handoff-PNG есть, он ПОЛНОСТЬЮ заменяет процедурное небо (звёзды/город/вышку), а не
  накладывается поверх него. Если одновременно убрать альфу (сохранить как RGB),
  decoded-память при увеличении размера не вырастет, а даже чуть упадёт:
  сейчас 1672×941×**4** байта (RGBA) ≈ 6.0 МБ → новый 2000×1000×**3** байта (RGB)
  ≈ 5.7 МБ. Крупнее и чётче — весит столько же или чуть меньше. Предварительно стоит
  проверить в редакторе, что в файле действительно нет полупрозрачных пикселей.
- **`sprite_apron.png` (1672×941, уже RGB без альфы) — запас на срез небольшой.**
  Даже на умеренном ориентире (планшет/Chromebook 1280×720, `dpr=2`, `ui≈1.16`)
  нужно **≈1304×944** — практически совпадает с нынешней высотой (944 vs 941, резать
  некуда), запас есть только по ширине. **Рекомендуемая цель: 1304×944** (−22% по
  ширине) → decoded-память ≈6.13 → ≈4.9 МБ (**~20%**, заметно меньше, чем у
  `svc_*`/`plane2`). Ниже приоритетом, чем остальные пункты этой таблицы — соотношение
  риска (софт на планшетах) к выигрышу здесь хуже.
