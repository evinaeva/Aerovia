# Firebase Remote Config — фиче-килсвитч + поэтапный релиз контента

Оба механизма живут в одном JS-слое — [`src/game/12h-remote-config.ts`](../src/game/12h-remote-config.ts) —
и тянутся одним нативным `fetchAndActivate` (плагин `RemoteConfigPlugin`, заводится в
[`scripts/setup-android.mjs`](../scripts/setup-android.mjs)). На вебе/PWA плагина нет → игра остаётся на
запечённых в сборку дефолтах. **Ничего из этого не требует пересборки/переливки APK — всё правится в
консоли Firebase → Remote Config.**

Две группы параметров:

| Параметр (RC) | Тип | Что делает |
|---|---|---|
| `survival_leaderboard` | Boolean | Killswitch глобального рейтинга (экран/кнопка/отправка счёта/ранг-виджет). Дефолт `true`. |
| `new_achievements` | Boolean | Killswitch соревновательных ранг/сезон-медалей. Дефолт `true`. |
| `content_flags` | **JSON** | Поэтапный релиз уже готового контента (ниже). Дефолт — всё закрыто. |

---

## `content_flags` — поэтапное открытие контента (staged rollout)

Один **JSON**-параметр Remote Config. Управляет ВИДИМОСТЬЮ уже реализованного контента — не разработкой.
Форма:

```json
{
  "bonus_levels":  { "enabled": false, "unlock_at": "2026-08-15T00:00:00Z" },
  "survival_mode": { "enabled": false, "unlock_at": "2026-09-01T00:00:00Z" },
  "biomes_pack_2": { "enabled": true,  "unlock_at": null },
  "season_league": { "enabled": false, "unlock_at": null }
}
```

### Ключи и что они гейтят

| Ключ | Что открывает |
|---|---|
| `bonus_levels` | Бонус-уровни (ячейки «N½» в карте уровней, после 5/10). Пока закрыт — «coming soon» вместо прогресс-замка. |
| `survival_mode` | Режим Survival целиком: кнопка «Выживание» на старте + базовый набор биом-карт (forest / arctic / tropical). |
| `biomes_pack_2` | Второй набор биом-карт (desert / mountain / megacity). Требует также открытого `survival_mode`. |
| `season_league` | Лига сезона: чип-отсчёт на старте, вкладка «Сезон» в рейтинге, начисление дивизионов/призов. Требует также `survival_leaderboard`. |

### Как разрешается флаг (приоритет)

Для каждого ключа игра решает «открыт ли контент сейчас» так:

1. **`enabled: true`** → открыт **сразу**, дату игнорируем (ручной оверрайд — «открыть прямо сейчас»).
2. иначе **`unlock_at` задан и время наступило** (текущее UTC-время устройства ≥ `unlock_at`) → открыт **по расписанию**
   (даже если `enabled` остался `false` в конфиге — так старая версия приложения откроет контент вовремя).
3. иначе → **закрыт**, показывается плашка «Coming soon» (с датой, если `unlock_at` задан).

> `unlock_at` — ISO 8601 в **UTC** (`…Z`). `null` = «без плановой даты» (только ручным `enabled`).

### Типовые операции в консоли (без пересборки)

- **Открыть контент по плану:** задать `unlock_at` на нужную дату (UTC), `enabled` оставить `false`.
  Наступит дата — откроется у всех, включая офлайн-устройства (по их часам).
- **Открыть прямо сейчас (сдвинуть план вперёд руками):** поставить `enabled: true`. Дата больше не важна.
- **Передвинуть дату:** поменять `unlock_at`. Раньше времени — закроется обратно; позже — откроется позже.
- **Экстренно закрыть уже открытое (killswitch):** поставить `enabled: false` **и** `unlock_at: null`.
  Оба условия открытия ложны → контент прячется у всех.

После сохранения в консоли: клиенты подтянут новый конфиг при следующем `fetchAndActivate` (старт приложения /
периодический minimumFetchInterval), UI сам перечитает гейты (событие `pf:flags`).

### Поведение офлайн / без свежего конфига (безопасный fallback)

- Firebase SDK **кэширует последний удачный удалённый конфиг** на устройстве и отдаёт его офлайн — так что
  единожды применённое расписание/оверрайд переживает потерю сети.
- Если удалённый конфиг **ни разу** не активировался (свежая установка без сети, или веб-версия), игра
  использует **запечённые в сборку дефолты** — сейчас это **всё закрыто** (`enabled:false`, `unlock_at:null`),
  т.е. весь staged-контент показан как «coming soon». Это осознанный безопасный fallback: не «всё открыто
  вслепую». Факт работы на дефолтах логируется в аналитику (`remote_config`, `usingDefaults`).
- Предсказуемость между версиями: и свежая, и недельной давности сборка, подтянув один и тот же
  `content_flags`, разрешают открыт/закрыт **одинаково** (логика детерминирована от конфига + часов).

### Дев-превью

В дебаг-меню (Настройки → Debug) есть галочка **«Открыть все coming soon»** (`debug.unlockContent`) —
локально открывает весь запертый staged-контент разом, минуя даты/флаги. Не трогает удалённый конфиг,
хранится в `localStorage` рядом с прочими читами.

## Что нужно один раз завести в консоли Remote Config при следующем релизе

1. **Boolean** `survival_leaderboard` = `true` (если ещё не заведён).
2. **Boolean** `new_achievements` = `true` (если ещё не заведён).
3. **JSON** `content_flags` — значение по образцу выше. Стартовый безопасный вариант — всё закрыто:
   ```json
   { "bonus_levels":{"enabled":false,"unlock_at":null},
     "survival_mode":{"enabled":false,"unlock_at":null},
     "biomes_pack_2":{"enabled":false,"unlock_at":null},
     "season_league":{"enabled":false,"unlock_at":null} }
   ```
   Затем открывать волнами: проставлять `unlock_at` (расписание) или `enabled:true` (сразу) по мере релиза.

Ключи и дефолты — единственный источник истины в коде ([`12h-remote-config.ts`](../src/game/12h-remote-config.ts));
нативный плагин их только регистрирует и присылает активные значения обратно.
